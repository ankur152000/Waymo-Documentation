This user manual is designed to support Operators, Process Engineers, and End Users in effectively setting up, operating, and maintaining the station software and tooling environment. Waymo Automation Tool Architecture and Design Document

  1. Introduction

    1. Overview of the Waymo Automation Tool

      The Waymo Automation Tool is a robust and extensible cross-platform system designed to enable seamless integration, configuration, control, and monitoring of hardware devices used in automated manufacturing and testing environments. Developed with a modern microservice architecture, Waymo facilitates smooth communication between software components, embedded drivers, and a unified user interface, making it a powerful solution for industrial automation.

      At its core, Waymo is built on .NET Core 8.0, ensuring cross-platform compatibility and high-performance execution across Windows 10 and Windows 11 systems. It uses gRPC for client-server communication, which provides efficient binary serialization and supports multiple streaming options (unary, client streaming, server streaming, and bidirectional streaming), making it significantly faster and more secure than traditional HTTP REST APIs.

    2. Purpose of the Document

      This document provides a comprehensive and detailed description of the architecture and design of the Waymo Automation Tool Solution. It covers the hardware components, software modules, and their interactions, including the Human-Machine Interface (HMI) layer.

      It also outlines the rationale behind key design decisions, such as the choice of:

      • Operating systems
      • Hardware platforms
      • Communication interfaces
      • Software frameworks
      The goal is to present a holistic overview of the system and the reasoning behind its architectural choices.

  2. Problem Statement

    In the development and testing environment at Waymo, a wide variety of hardware devices are involved. These devices pose a challenge due to:

    • Diverse driver compatibility:
      • Some hardware supports only Linux drivers
      • Others are compatible only with Windows drivers
    The central challenge is to establish seamless communication and control over all these heterogeneous devices via a single, unified HMI application.

  3. HMI Application Requirements

    1. Drag-and-Drop Interface for Test Case Creation

      A user-friendly HMI enables users to visually create test cases by dragging and dropping devices

      • The HMI offers a visual interface that enables users to:
        • Define input-output relationships between components
        • Drag and drop devices and controls
      This simplifies test case creation and allows even non-technical users to configure hardware interaction logic easily.

    2. User Privileges and Access Control

      The system implements multi-level user privileges, offering role-based access for Administrators, Technicians, Engineers, and Operators—each with defined permissions for creating, executing, or reviewing use cases and reports.

      • The application supports multiple user roles such as:
        • Administrator
        • Technician
        • Engineer
        • Operator
      • Each role has specific privileges, including:
        • Creating/modifying use cases
        • Executing use cases
        • Viewing logs and reports
      This ensures both flexibility and security in a collaborative environment.

    3. Test Case Execution and Reporting

      Users can execute defined test cases directly from the interface, with real-time monitoring and automated generation of detailed logs and reports for traceability, validation, and continuous improvement.

      • Users can:
        • Execute test cases with a single command
        • Monitor performance and behavior
        • Generate detailed reports for each session
      These reports help track test success/failure and evaluate hardware performance across different conditions.

  4. Key Technologies and Architecture Overview

    This architecture illustrates a cross-platform automation framework for managing and controlling devices in a secure and modular way, using Node.js, gRPC, React, C#, Python, and .NET Core.



    View

    The Following are the core architectural components of the Waymo Automation Tool

    1. .NET Core 8.0 (LTS)-Platform Layer

      The system supports both Windows and Linux operating systems, with command-line access provided for executing backend scripts and managing processes. The Waymo service is built using the .NET Core 8.0 framework, enabling true cross-platform compatibility—allowing the application to run seamlessly across different operating systems, browsers, and device types.

    2. Node.js Environment

      • Acts as the central web backend for Waymo UI
      • Hosts:
        • React Frontend – UI for user login, test management, and reports.
        • Node-RED – A visual tool to manage device create flows, flow editor, recipe rules, and test cases.
        • Handles Security Encryption and forwards secure messages via gRPC.
        • Node JS is a cross-platform, open-source JavaScript runtime environment that can run on Windows, Linux, UNIX, macOS, and more.
        • Node environment has built for the node-red and react development and currently, we are using node version 20. The Node-red and React applications both will be running inside node environment.

    3. gRPC Communication

      • Enables structured, high-performance communication between:
        • Node.js environment (front end/back end)
        • C#/.NET application (backend and driver layer)
      • Ensures real-time control and data exchange with typed protobuf messages.
      • Communication used over client and server is gRPC, which much faster than Http REST since serialization happens in binary format compared to JSON/XML for that of REST. GRPC framework supports following types of RPC
        • Unary call
        • Client Streaming
        • Server Streaming
        • Bi-Directional Streaming

    4. .NET Core & C# Backend

      • Core logic for:
        • Device Repository – Tracks all connected device information.
        • Service Monitor – Monitors health/status of device drivers.
        • Logging – Stores diagnostics and errors from devices.
        • Priority Queue – Schedules device tasks based on importance.

    5. Node-RED in Waymo Automation Project:

      Node-RED is a powerful, flow-based, low-code development tool designed for visual programming, particularly well-suited for integrating hardware devices, APIs, and online services, often used in Internet of Things (IoT) solutions.

      • Integrated within the Node.js environment, it works alongside the React frontend.
      • Used for:
        • Recipe/Rule Engine: Defining and controlling test conditions or workflows.
        • Test Management: Orchestrating the start/stop of test sequences and collecting results.
      • Interfaces with backend services via secure gRPC communication, enabling real-time updates and robust control mechanisms.

    6. Device Drivers

      Device drivers are developed in Python, C++, or C to interface with specific hardware components such as robots and sensors. Drivers written in C++ and C are dynamically loaded into the .NET backend using DLLs, while Python drivers are integrated using Python.NET

    7. SQLite Database

      SQLite is used as a local, lightweight relational database for storing essential data without the need for a separate server process.

      In Waymo Automation Tool, SQLite Database is used for:

      • Device data
      • Logs
      • User settings
      • Job/test configurations

    8. Device Repository

      The device repository is the C# class library where all the business logic sits. Device Drivers has activated inside the device repository for the API call. It has capturing of errors and information and logs them into the SQLite Db and Log files on the system.

    9. Python.NET

      Python.net has used as an interface to exchange data between the Python drivers and .NET

    10. DLL Import

      DLL import has used as an interface to exchange data between C++ drivers and .NET

    11. C++ Drivers

      C++ drivers are nothing but C++ project libraries created in the waymo solution for some devices. Device such as Camera, Motor has written in C++, as those SDK’s are compatible with it.

    12. Python Drivers

      Python drivers are nothing but python scripts written and added in the waymo solution for some devices. Device such as Janome, Brainbox has written in python as those SDK’s are compatible with it.

    13. Device Support and Drivers

      Waymo supports multiple SDKs and integrates them via:

      • C++ Drivers
      • C Drivers
      • Python Drivers

  5. Major Modules in Waymo Automation Tool

    1. Device Management

      In device management, the system will have to provision to control each device minimum functionality from the UI and will have the provision to add/update new/existing device details along with search feature. Device management will have complete control on individual devices with all its minimum functionality. System will have the below features under device management

      • Add or update device configurations

    2. User Role Management

      In user role management, system will have the provision to create user and roles along with the permissions mapping to each role. This will restrict the access of the user to specific screens in the system as per the user role mapping configurations. System will have the below features under user role management

      • Role-based access control
      • Create users, Create Role, Create Role and Permission Mapping
      • Screen-level restrictions based on role
      • Create User Role Mapping

    3. Rule/Recipe Engine (Node-RED)

      This module has integrated completely in Node-RED. In system, recipes have created with multiple ingredients. Each ingredients is created as reusable node block or sub flow in node-red. In node-red category has created for each device. N number of recipes has listed inside single node-red dashboard (UI). Node-RED has integrated into the React Application where only process engineer can access the UI based on user role mapping. This module will have the below features on Node-RED Platform.

      • Create Reusable flows.
      • Create Functions
      • Create Dashboard
      • Create a Category for New Devices
      • Configure global variables for recipe creation

    4. Dashboard

      This module has been integrated inside the Node-RED Platform, where user can execute the recipe and view the results on the dashboard controls.

      • Trigger recipes and monitor results

      • Provide automation process snapshots

    5. Test Management

      Test management will have test runs on each station with type as dev, staging, debug and production. It is been integrated into the node-red platform.

      • Run tests in different environments (dev/bug fix/production)
      • Monitor/log test behaviour at station level

    6. Error Logging

      Errors are captured in the log file and in the SQLite DB. System tray icon has the option to view a log, which take user to the file location. Error log to be captured at the node-red platform

      • Capture .NET and device driver errors
      • Store in log files and SQLite
      • View via system tray log viewer

    7. Reports

      The reports module has to be integrated into the React application, where all reports will be displayed

      • React-based module
      • Shows device usage, test results, and recipe logs

    8. System Tray

      The Waymo service is designed to reside in the system tray, offering easy accessibility and control. Upon restarting Windows, the Waymo service is configured to start automatically, ensuring uninterrupted operation and reducing manual intervention. Following

      • start node-red
      • stop node-red
      • start gRPC
      • stop gRPC
      • exit
      • devices online

    9. Security & Encryption

      GRPC uses HTTP/2 as its transport layer protocol to communicate. There are actually two types of checks happening on the client side: certificate check and hostname verification check. The former checks the server certificate to make sure the client trusts it; the latter checks the target name with the server's identity on the peer certificate. The server uses Https redirection where the server validates the certificate sent by client and then redirects the request. gRPC runs over HTTP/2 with TLS encryption, including:

      • Server certificate validation
      • Hostname verification
      • HTTPS redirection
      These ensure secure and trusted client-server interactions

    Conclusion

    The Waymo Automation Tool offers a comprehensive, modular, and future-ready solution for managing device interactions in industrial and test environments. With its microservice design, visual flow-based programming, secure communication, and user-centric dashboards, Waymo provides a scalable and high-performance platform tailored to evolving automation needs.

Effective UI troubleshooting ensures a seamless and intuitive user experience, improves usability, and helps maintain the overall quality and reliability of the application.

Emergency Stop and Recovery Process

  1. Purpose

    This document outlines the standard procedures for initiating an emergency stop (E-Stop) and safely recovering tool or system operations after an emergency halt. It is intended to protect operators, equipment, and surrounding environments from harm or damage.

  2. Emergency Stop Procedure

    • When to Trigger Emergency Stop

      • Initiate an Emergency Stop immediately in the following cases:

      • Risk of injury to personnel
      • Sudden, unintended tool movement
      • Abnormal sound, vibration, or smoke
      • Malfunctioning safety interlocks
      • Electrical short circuit or sparks
      • Fire, fluid leakage, or hazardous gas release

    • How to Activate

      1. Locate the Emergency Stop Button or Switch
        • Clearly marked in red with a yellow background
        • Typically mushroom-style pushbutton
      2. Press the Emergency Stop Button Firmly
        • Tool/system should immediately cease operation
        • Power may be cut depending on system design
      3. Evacuate if Necessary
        • If risk to health or safety is present, follow site evacuation procedures
      4. Report the Incident
        • Inform supervisor or safety officer immediately
        • Fill out the Incident/Stop Report Form (see Section 6)

  3. Recovery ProcedureAuthorized Personnel Only

    Only trained and authorized technicians may perform recovery procedures.

    1. Perform Safety Assessment
      • Inspect the tool/system for visible damage or hazards
      • Ensure the cause of the stop is understood and resolved
    2. Reset Emergency Stop Button
      • Turn or pull the E-Stop button (as designed) to release
    3. Restore Power/System State
      • Re-enable system power if disconnected
      • Restart relevant subsystems (motors, sensors, network)
    4. Run Diagnostics or System Check
      • Use control software or onboard diagnostics
      • Check logs or alerts for error codes
    5. Test in Safe Mode
      • Operate tool/system in a restricted or manual mode to verify safety
    6. Resume Normal Operation
      • Only after all checks pass and supervisor gives approval

Introduction to Node-RED

  1. Introduction

    Node-RED is a flow-based development tool for wiring together hardware devices, APIs, and online services. Node-RED provides an easy-to-use visual interface for creating applications that interact with IoT devices, APIs, and web services. It is designed to simplify integration between various hardware and software, making it a popular choice for IoT, automation, and data-flow applications.

    1. What is Node-RED?

      Node-RED is an open-source, browser-based flow editor that enables users to connect devices, APIs, and online services in a seamless and visual manner. It uses a flow-based programming model where logic is constructed by wiring nodes together in a flow chart-like design. These nodes can represent hardware devices, API endpoints, databases, or even custom functions.

      Node-RED's easy-to-use interface and wide range of supported nodes make it an excellent choice for both beginners and advanced users to quickly create powerful applications without extensive programming knowledge.

    2. Key Features of Node-RED

      • Flow-based programming:The primary concept behind Node-RED is the flow-based programming paradigm. Users can create applications by connecting pre-configured nodes (blocks of code) to form logical sequences.
      • Drag-and-drop Interface:With its intuitive, web-based flow editor, Node-RED allows users to drag and drop nodes to quickly build applications.
      • Extensibility:Node-RED has a rich ecosystem of nodes available through the Node-RED library. You can add nodes for different communication protocols (HTTP, MQTT, Web-Sockets, etc.), data storage (SQL, NoSQL), and even machine learning (TensorFlow, Watson).
      • Integration:Node-RED makes it easy to integrate with IoT devices, third-party services, and cloud platforms such as AWS, IBM Watson, and Google Cloud. It supports a variety of communication protocols like MQTT, HTTP, Web-Sockets, and more.
      • Real-time monitoring and debugging:You can monitor, debug, and log data flows in real-time, making troubleshooting easier during development.
      • Custom Functions:For advanced users, Node-RED allows you to write custom JavaScript functions, providing flexibility, code re-usability and used for more complex use cases.

    3. Components of Node-RED:

      Node-RED is composed of three main components:

      • Flow Editor:This is the user interface where you design and configure flows. You can drag nodes from the left panel, connect them, and deploy them to the runtime. The flow editor runs in a web browser.
      • Nodes:Nodes are the building blocks of your application in Node-RED. They represent the various components that you can interact with, such as devices, services, or custom functionality. Node-RED provides pre-built nodes for common tasks like HTTP requests, MQTT messages, and GPIO (General Purpose Input/Output) operations.
      • Example nodes include:
        • Inject Node:Used to generate messages manually or on a schedule.
        • HTTP Request Node:Used to make HTTP requests to external services.
        • Function Node:Allows you to write custom JavaScript to process data. Similarly we have multiple nodes in node-red dashboard with its variety of nodes which is used to develop UI dashboard also.
      • Runtime:Node-RED runs on a server (local or remote), processing the logic defined in the flow editor. The runtime is responsible for executing the flows, listening for incoming events, and interacting with the connected devices or services.

  2. Setting Up Node-RED

    1. Installation:

      • Node-RED is built on Node.js and can be easily installed using npm (Node.js package manager).
      • To install Node-RED globally, run the following command in your terminal:

        • Bash or terminal

        npm install -g --unsafe-perm node-red

    2. Starting Node-RED:

      • After installation, you can start Node-RED by running:

        • Bash or terminal

        node-red

      • This will start the Node-RED service and open the flow editor at http://localhost:1880 in your web browser.

    3. Accessing the Flow Editor:

      • After starting Node-RED, you can access the flow editor by navigating to http://localhost:1880 in your browser.
      • You can begin creating flows by dragging nodes from the left sidebar and connecting them to form your application.

  3. Creating Your First Flow:

    1. Inject Node:

      To start, drag an Inject Node onto the flow editor canvas. This node will trigger an event, such as sending a message.

    2. Debug Node:

      Drag a Debug Node to the canvas. This node will output the results of the flow to the debug window.

    3. Connecting Nodes:

      Connect the Inject Node to the Debug Node by clicking the small squares on the sides of the nodes and dragging a line to connect them.

    4. Deploy the Flow:

      Click on the Deploy button in the top-right corner to activate your flow. After deployment, when you trigger the Inject Node, the Debug Node will log the result in the debug window.

    5. Test Your Flow:

      Click on the square of the Inject Node to trigger the flow. You should see the output appear in the debug tab on the right side of the editor.

  4. Working with External Services:

    One of Node-RED's strengths is its ability to integrate with various external services. For example, you can use an HTTP Request Node to make API calls to third-party services and can perform CRUD operations.

  5. Extending Node-RED with Additional Nodes:

    Node-RED has a large library of community-contributed nodes available on the Node-RED Library. To install additional nodes, you can go to the Manage palette option in the editor and search for new nodes or install them via npm commands.

Node-RED Execution and Flow Troubleshooting Guide

  1. Introduction

    Node-RED is a flow-based development tool where each node performs a specific task and passes the message (msg) to the next node. Below is a breakdown of flow behavior and lifecycle.

    1. Message Flow:

      • A flow begins with an input node (e.g., HTTP In, MQTT In, Inject).
      • The message flows through processing nodes (e.g., Function, Change, Switch).
      • It ends in output nodes (e.g., HTTP Response, MQTT Out, Debug).

    2. Execution Lifecycle:

      • Trigger:A flow is triggered by an event (e.g., Inject button, HTTP request).
      • Propagation:The msg object is passed from one node to the next via wires.
      • Processing:Each node processes the msg according to its logic.
      • Termination:The flow ends with a node that does not pass data further (like HTTP Response).

    3. Execution Context:

      Node-RED provides three types of data context:

      • msg: Per-message scope. Changes only apply to the current message.
      • flow: Shared across nodes within the same flow.
      • global: Accessible across all flows in the instance.

  2. Common Issues and How to Resolve Them

    1. Issue 1: Node Not Executing:

      Cause: Node conditions not met or previous node not passing data

      How To Fix

      • Use Debug nodes after each node to see if msg is passed.
      • Check wires to ensure nodes are connected properly.

    2. Issue 2: Undefined or Empty msg Properties

      Cause: Accessing properties that do not exist.

      How To Fix

      • Add checks in Function nodes:

        • if (!msg.payload) {
        node.error("Payload is missing", msg);
        return null;
        }

    3. Issue 3: Function Node Not Working

      Cause: Syntax errors or logic issues.

      How To Fix:

      • Check Node-RED logs for errors.
      • Use node.warn() and node.log() to trace.

    4. Issue 4: Inject Node Doesn’t Trigger

      How To Fix:

      • Make sure node is deployed
      • Try manually clicking the Inject button.

    5. Issue 5: Flows Not Deploying

      How To Fix:

      • Ensure there are no syntax errors in Function nodes.
      • Check if any required nodes are missing (e.g., custom nodes).

  3. Advanced Debugging Techniques

    1. Use the Debug Panel

      • Set Debug nodes to show the complete message.
      • Place Debug nodes strategically across flows.

    2. Logging in Function Nodes

      node.warn("This is a warning");
      node.log("This is a log message");

    3. Catch and Status Nodes

      • Use Catch node to capture and log errors.
      • Use Status node to monitor node status (e.g., connected, error).

  4. Flow Management

    1. Flow Tabs

      • Organize logic across multiple flows.
      • Use Link In/Out nodes to manage flow transitions.

    2. Backups

      • Always keep exported JSON copies of important flows.
      • Use version control (Git) for production flows.

    3. Flow Configuration

      • Use Environment Variables for dynamic values.
      • Use Context Storage to persist data across deploys.
      • Leverage Credentials storage for secrets like API keys.

  5. Node-RED Runtime Troubleshooting

    1. Understanding Runtime Architecture

      • Node-RED runs on Node.js.
      • Each flow is compiled to a set of JavaScript functions.
      • Nodes run sequentially, and asynchronously if needed.
      • Runtime maintains a single-threaded event loop but leverages async I/O.

    2. Accessing Logs

      • Use terminal:

        node-red-log

      • Or check logs in

        /var/log/node-red.log

    3. Restart Node-RED

      • Run Command in terminal

        sudo systemctl restart nodered

      • Checking Port Usage

        netstat -tulnp | grep :1880

    4. Performance Monitoring

      • Use top, htop, or Node-RED’s built-in metrics (if enabled).
      • Consider enabling Node-RED Flow Metrics to track throughput and delays.

  6. Debugging Custom Nodes

    • Ensure the node is installed properly using npm.
    • Restart Node-RED after installing custom nodes.
    • Check the node’s source code in ~/.node-red/node_modules/
    • Use try/catch inside the custom node function for error handling.

  7. Troubleshooting Flows

    1. Tips for Diagnosing Flow Issues

      • Use Debug Nodes generously to trace the path of msg.
      • Validate logic in Switch and Change nodes.
      • Inspect Function nodes for proper output.
      • Review flow triggers — some flows may not trigger without specific conditions.

    2. Ensuring Flow Reliability

      • Avoid hard-coding paths and values.
      • Use environment variables or configuration nodes.
      • Group common patterns into subflows for reusability.

    3. Preventing Flow Errors

      • Always check if the required properties exist in msg.
      • Wrap logic in try/catch blocks if using advanced scripting.
      • Use return null; to stop flows when needed.

    4. Handling External APIs or Inputs

      • Add Retry/Timeout mechanisms.
      • Use Status and Catch nodes to handle unexpected failures

GLOSSARY

  • 3D Laser Profiler : A device used for measuring surface profiles and shapes in three dimensions using laser technology.
  • LJ-V Series : A series of 3D laser profilers manufactured by Keyence, including models like LJ-V7000, LJ-V7300, LJ-V7500, etc.
  • Profile : A cross-sectional view of the surface of an object captured by the 3D laser profiler, showing height or depth information.
  • Measurement Range : The distance over which the 3D laser profiler can accurately measure surfaces.
  • Resolution : The smallest change in height or depth that the 3D laser profiler can detect.
  • Repeatability : The ability of the 3D laser profiler to produce consistent results when measuring the same object multiple times.
  • Sampling Interval : The distance between individual measurement points along the profile.
  • Point Cloud : A set of data points in three-dimensional space representing the surface of an object, captured by the 3D laser profiler.
  • Calibration : The process of adjusting the settings of the 3D laser profiler to ensure accurate measurements.
  • Alignment : The process of positioning the 3D laser profiler relative to the object being measured to obtain the desired measurement results.
  • Ethernet/IP : A communication protocol used for connecting the 3D laser profiler to other devices or systems in an industrial automation environment.
  • Measurement Tools : Software features of the 3D laser profiler used for analyzing profiles, such as height measurements, area calculations, and defect detection.
  • Stable Run : A mode in which the 3D laser profiler continuously monitors and adjusts its settings to maintain stable and accurate measurements.
  • Trigger Mode : A mode in which the 3D laser profiler captures a profile only when triggered by an external signal, allowing for precise control of measurement timing.
  • Surface Flatness : A measurement of how flat or smooth a surface is, typically expressed as a deviation from a perfectly flat plane.
  • Height Deviation : The difference in height between a point on the surface and a reference plane, used to assess surface roughness or profile irregularities.
  • Dynamic Range : The range of surface heights that the 3D laser profiler can measure, from the lowest to the highest point.
  • Deflection : The bending or deformation of an object's surface, measured by comparing its profile to a reference profile.

GLOSSARY

  • Gocator: A family of 3D smart sensors manufactured by LMI Technologies, used for inline inspection, measurement, and quality control in industrial applications.
  • 3D Point Cloud: A set of data points in a three-dimensional coordinate system that represents the external surface of an object scanned by a 3D sensor, such as the Gocator.
  • Field of View (FOV): The area that a sensor can capture or observe at any given moment. It is typically defined by the sensor's lens and sensor size.
  • Resolution: The smallest detectable change in position or dimension that a sensor can measure. Higher resolution sensors can detect smaller features.
  • Scan Rate: The speed at which a sensor captures and processes data. It is usually measured in Hz or frames per second (FPS).
  • Measurement Range: The range of distances over which a sensor can accurately measure objects. It is typically defined by the sensor's optics and laser.
  • Profile: A 2D cross-section of a 3D object captured by the sensor. Gocator sensors often capture profiles along a line to build a 3D point cloud.
  • ROI (Region of Interest): A specific area within the sensor's field of view where measurements or inspections are performed.
  • Calibration: The process of adjusting a sensor to ensure that its measurements are accurate and consistent. Calibration is often necessary to correct for distortions or misalignments.
  • Alignment: The process of adjusting the sensor's position or orientation to ensure that it is correctly aligned with the object being scanned.

    How Our Driver Works with Virtual Status Outputs

    Our Python driver for the Banner SC10 safety controller was specifically designed to interface with the controller’s Virtual Status Outputs, or VSOs. These software-defined register status bits are configured within the SC10 project (.xsc) file. VSOs act as named indicators for system conditions such as E-stops, gate switches, relays, etc.

    What does the driver do?

    1. It parses the .xsc to identify all defined VSOs.
    2. For each VSO, it reads the relevant Modbus registers from the controller.
    3. It maps each VSO number to its human-readable name, presenting statuses like “E-STOP HIGH” or “Gate1 LOW” in the application.
    4. This enables instant diagnostics, monitoring, and integration—without manual bit mapping for each physical input or output.
    5. A critical factor here is that Modbus registers only provide raw IO line status (ON/OFF levels), not the names of the connected devices. This is an inherent software limitation of the SC10.
    6. Without VSOs, all we get are ambiguous bits (e.g., “Register 40001 bit 3 = HIGH”) with no clue whether that represents an E-Stop, a Gate input, or a Safety relay.
    7. Operator visibility and troubleshooting become severely impaired.
    8. The only practical way to achieve readable mappings is through VSOs in the project configuration.

    Steps to Add VSOs to Your SC10 Project

    To restore full functionality and diagnostic clarity, please update your Banner Safety Controller project as follows:

    How to add VSOs in Banner SC10 Configuration Software:

    1. Open Banner SC10 Project in the configuration software.
    2. Navigate to the Virtual Status Outputs section.
    3. Add a new VSO for each safety input/output/function block you need to monitor. Assign a meaningful name (e.g., "E-STOP", "Gate1", "Relay1").
    4. Link each VSO with the correct input, output, or logic point. This ensures their bits track real system events.
    5. Save and export the updated .xsc configuration.
    6. Upload the configuration to SC10 device as per normal procedure. Restart the Controller.
    7. Share the updated .xsc file. I will recheck it again

    This process does not require any hardware changes—VSOs are purely a software configuration.

    Note:How the Controller Detects Malfunctions

    1. To detect malfunctions the controller monitors safety inputs, such as limit switches, and internal safety conditions, such as comparing the reference position with the software limits.
    2. Internal safety conditions may consist of a static formula, such as checking the acceleration limit, or may include time dependencies, such as measuring the time interval between two interrupts to detect servo interrupt faults.
    3. Some safety conditions are a mixture of both techniques. For example, position error control is based on a static condition, whether a motor is positioned at a certain location, and a time dependency, how long the motor is positioned at the location.
